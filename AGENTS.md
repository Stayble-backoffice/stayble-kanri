# AGENTS.md

このリポジトリで Codex が最初に読む指令塔です。  
株式会社Staybleの住宅宿泊管理LP「乗り換え民泊管理」を制作するためのルールを定義します。

---

## 1. Codex の役割

あなたは、株式会社Staybleの住宅宿泊管理LPを制作する実装担当です。  
目的は、単なる綺麗なLPではなく、**開業済み民泊オーナーが既存管理会社から乗り換えたくなるWebサイト**を作ることです。

このLPの中心思想は以下です。

> 今の管理会社、高すぎませんか？  
> 清掃会社が管理を請け負う時代へ。

Staybleは「これから民泊を始めたい人」向けの開業支援会社ではありません。  
すでに民泊を運営していて、管理費・清掃品質・リネン・現地対応に不満があるオーナーの乗り換え先です。

## 3. 最重要コンセプト

Staybleの住宅宿泊管理LPは、以下の定義を軸にする。

> すでに開業済みの民泊オーナー向けに、既存管理会社からの乗り換え先として提供する低コスト型の民泊管理サービス。

売るものは「開業支援」ではない。  
売るものは、**管理費を抑えながら、清掃・リネン・現地対応までちゃんと回す乗り換え管理**です。

---

## 4. サービス名

正式なサービス名は以下。

```txt
乗り換え民泊管理
```

LP内では以下のような言い換えはOK。

- 開業済み民泊向けの乗り換え管理
- 管理会社からの乗り換え先
- 清掃会社がつくる民泊管理
- 清掃・リネン込みの民泊管理

---

## 5. 料金表現の絶対ルール

最重要。

```txt
管理費 ＝ 月額固定費5,000円〜 + 売上歩合5%
```

正しい表現:

- 「月額5,000円〜 + 売上5%」
- 「管理費は、物件規模別の月額固定費と売上歩合5%で構成されます」
- 「管理費（月額固定費 + 売上5%）」

禁止表現:

- 「管理費5%」
- 「5%の管理費」
- 「管理費は売上の5%のみ」

固定費部分が消える表現は絶対にしないこと。

---

## 6. 実績表現の絶対ルール

既存資料の実績は、**清掃管理事業の実績**です。  
住宅宿泊管理業としての実績に見せてはいけません。

OK:

- 「民泊清掃管理で約240室」
- 「年間清掃件数 約15,000件」
- 「清掃会社として培った現場運用力」

NG:

- 「住宅宿泊管理で約240室」
- 「管理代行実績240室」
- 「管理業として年間15,000件」

---

## 7. LPで前面に出すこと

- 今の管理会社が高いと感じるオーナーへの問題提起
- 乗り換え民泊管理
- 月額5,000円〜 + 売上5%
- 初期費用0円
- 写真撮影0円
- OTA初期立ち上げ0円
- 清掃・リネンはStaybleセット
- 清掃会社が管理を請け負う合理性
- 手取りシミュレーション
- 清掃品質まで責任を持つこと
- 開業済み物件向けであること

---

## 8. LPで前面に出さないこと

- AI活用
- WindWoodsなど別ブランド名
- 違約金
- 契約解除条件
- ゴミ回収を強く宣伝する表現
- 住宅宿泊管理業登録番号の捏造
- 実在しない導入事例やお客様の声
- 管理業としての未確定実績

---

## 9. ロゴ

ユーザーはリポジトリ直下にロゴ画像を配置済み。  
Codexは、リポジトリ直下にあるロゴ画像を優先して使用すること。

候補ファイル名:

```txt
logo.png
logo (1).png
stayble-logo.png
```

実装時は、存在するロゴファイルを確認してから参照すること。  
ロゴを勝手に再生成しない。

---

## 10. サイト構成

サービスは1本化するが、SEOと検索意図に合わせてページ分けを積極的に使う。

推奨URL:

```txt
/
/pricing/
/switching/
/services/
/cleaning-linen/
/emergency/
/area/
/faq/
/contact/
```

地域別ページの量産は初期では行わない。  
地域ページを作る場合も、地域名だけ差し替えたdoorwayページは禁止。

---

## 11. 技術方針

GitHub Pagesで公開しやすい静的サイトを優先する。

推奨:

- Astro + TypeScript
- Vite + React + TypeScript
- 純粋なHTML/CSS/JS

空リポジトリなら Astro を第一候補にする。
